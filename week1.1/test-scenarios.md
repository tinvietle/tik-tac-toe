# Tic-Tac-Toe Test Scenarios

| Title | Scenario Description | Priority | Notes |
| --- | --- | --- | --- |
| Startup Without Argument | Execute program with no CLI argument. Verify terminal prints exactly `Please, input a valid option [1-2]` and exits without starting game loop. | P0 - Critical | Validates mandatory entry condition. Also checks whether program should terminate or re-prompt (termination is implied, not explicit). |
| Startup With Invalid Argument | Execute with invalid argument values (for example: `0`, `3`, `-1`, `abc`). Verify exact message `Please, input a valid option [1-2]`. | P0 - Critical | Requirement defines invalid values but not whether extra whitespace or quoted values are accepted. |
| Startup With Extra Arguments | Execute with more than one argument (for example: `1 extra`). Verify expected handling (reject or ignore extras). | P1 - High | Missing functionality: behavior for extra arguments is not specified. |
| Startup Message and Order | Execute with valid start option (`1` then `2`). Verify output order is: `Hello!`, initial 3x3 board of zeros, then `Player#<n>'s turn`. | P0 - Critical | Order is strict in requirements and likely used by automated checks. |
| Initial Board Mapping | Confirm initial board has 9 cells and numbering mapping is top-to-bottom, left-to-right (1..9). | P1 - High | Ambiguity: numbering reference is described conceptually, but not explicitly required to be displayed alongside values. |
| Human Non-Integer Input | During Player#1 turn, input non-integer text (for example: `x`). Verify message `Please, input a valid number [1-9]`, then `Player#1’s turn`, and game waits for another input. | P0 - Critical | Uses curly apostrophe in requirement (`Player#1’s turn`) while earlier uses straight apostrophe (`Player#<n>'s`). Formatting inconsistency should be clarified. |
| Human Out-of-Range Integer Input | During Player#1 turn, input out-of-range values (`0`, `10`, `-5`). Verify same invalid-number flow as non-integer input. | P0 - Critical | Confirms validation path for numeric but invalid values. |
| Human Occupied Cell Input | Player#1 selects a cell already taken by either player. Verify `The cell is occupied!`, then `Player#1’s turn`, and no board change. | P0 - Critical | Ensure turn does not switch after invalid move. |
| Human Valid Move Updates Board | Player#1 inputs a free valid cell. Verify board updates selected cell to `1`, board is displayed, and game transitions to computer turn if not ended. | P0 - Critical | Core happy-path behavior. |
| Turn Handoff Symmetry | Verify strict alternation in both directions: with start option `1`, valid human move is followed by computer turn; with start option `2`, computer move is followed by human turn, then alternation continues. | P0 - Critical | Prevents skipped/double turns and explicitly covers both start modes. |
| Computer Move Strategy (First Free Cell) | After a valid human move, verify computer always picks the first available cell from `1` to `9`. | P0 - Critical | Deterministic strategy is explicitly required; useful for reproducible tests. |
| Computer Move Updates Board | Verify computer move writes `2` in chosen cell, displays board, and returns to human turn if game not ended. | P0 - Critical | Requirement does not mandate a dedicated "computer chose cell X" message; potential observability gap. |
| Human Win Detection | Prepare board where human winning move completes row/column/diagonal. Verify board print then `Player#1 won!` and program stops. | P0 - Critical | Must ensure no further prompts appear after win. |
| Computer Win Detection | Prepare board where computer winning move completes row/column/diagonal. Verify board print then `Player#2 won!` and program stops. | P0 - Critical | Must ensure win is checked immediately after computer move. |
| Draw Detection After Human Move | Prepare board where human takes last free non-winning cell. Verify board print then `It is a draw!` and stop. | P0 - Critical | Confirms terminal condition for full board without winner. |
| Draw Detection After Computer Move | Prepare board where computer takes last free non-winning cell. Verify board print then `It is a draw!` and stop. | P0 - Critical | Ensures draw logic is symmetric between players. |
| Win vs Draw Precedence on Last Cell | Set state where last available move both fills board and creates a win. Verify winner message appears (not draw). | P1 - High | Priority rule is implied by requirement ordering but not explicitly stated as precedence. |
| Board State Consistency Across Turns | Run a full game and verify each printed board reflects accumulated moves only: `1` for human, `2` for computer, `0` for empty. | P0 - Critical | Detects accidental resets or incorrect overwrites. |
| No Input Accepted After Game Ends | After win/draw, attempt additional input. Verify program has terminated and does not consume further entries. | P1 - High | End-of-program behavior is implied, not explicitly tested in requirements. |
| Input With Leading/Trailing Spaces | Enter values like ` 5`, `5 `, and blank line during human turn. Verify expected parsing behavior and error messages. | P2 - Medium | Ambiguity: whitespace handling is not specified. |
| EOF/Interrupted Input Handling | During human turn, send EOF (Ctrl+Z on Windows). Verify graceful handling (clean exit or clear error). | P2 - Medium | Missing functionality: requirements do not define behavior for input stream termination. |
| Board Rendering Format Strictness | Validate exact board layout per print (row separators, spacing, trailing spaces, newline count). | P1 - High | Output format details are underspecified; strict graders may require exact formatting. |
| OOP Structure Compliance Review | Review source design for OOP principles (separation of board, player, game loop responsibilities; encapsulation). | P2 - Medium | Non-functional requirement exists, but acceptance criteria are missing and subjective. |
| Localization/String Consistency | Verify apostrophe style and exact strings are consistent (`Player#1's` vs `Player#1’s`) across all outputs. | P1 - High | Ambiguity likely to cause test fragility in automated evaluation. |
